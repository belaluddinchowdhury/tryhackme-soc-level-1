# TryHackMe: SOC L1 Alert Triage — Writeup & Notes

This repository documents my analytical notes, core takeaways, and defensive methodologies implemented during the **SOC L1 Alert Triage** room on TryHackMe. This module establishes the structural framework required for a Junior Security Analyst to ingest security alerts, prioritize the log pipeline, perform investigative workflows, and execute accurate verdicts.

---

## 1. The Engineering Pipeline: From Events to Alerts
Inside a modern enterprise infrastructure, raw operational activities are generated by the millions. Identifying malicious anomalies requires a highly systematic data filtering loop:

[System Event] ──► [Log Generation] ──► [Telemetry Shipping] ──► [SIEM/EDR Engine] ──► [Actionable Alert]

* **System Event:** Any baseline programmatic action (e.g., user authentication, system process execution, or decentralized file downloads).
* **Log Generation:** Infrastructure layers (Operating Systems, network firewalls, cloud provider scopes) structurally index the metadata of the transaction.
* **Telemetry Shipping:** Automated log aggregators forward raw telemetry data from thousands of endpoints into a centralized repository.
* **The SIEM/EDR Layer:** Security Information and Event Management (**SIEM**) or Endpoint Detection and Response (**EDR**) architectures evaluate incoming logs against specialized detection rules. By highlighting only suspicious, anomalous behavior, it shields analysts from manual log reviews—transforming millions of raw logs into just dozens of high-priority **Alerts**.

## 3. Structural Properties of an Alert Payload

An execution block within a SIEM platform encapsulates critical properties that an analyst must evaluate before beginning a formal investigation:

1. **Alert Time:** The timestamp marking when the rule was triggered (typically experiencing a slight time-latency after the real event).
2. **Alert Name:** An operational summary of the suspicious activity compiled directly from the underlying detection rule rule-set.
3. **Alert Severity:** Urgency classifications (**Low, Medium, High, Critical**) indicating the potential impact to define tactical urgency.
4. **Alert Status:** The active lifecycle stage of the investigation loop (**New, In Progress, Pending, Closed**).
5. **Alert Verdict:** The conclusive analytical classification defining the validity of the hazard (**True Positive** or **False Positive**).
6. **Alert Assignee:** The active organizational ownership mapping identifying the defensive analyst handling the ticket.
7. **Alert Description:** The core logic profile explaining exactly why the detection rule tripped and its technical meaning.
8. **Alert Fields:** Granular indicators of compromise (**IOCs**) such as Source IPs, Hostnames, Target Hashes (MD5/SHA256), and Process Command Lines.

## 4. Triage Mechanics: Tactical Alert Prioritization

To manage active queues efficiently and mitigate critical threats in a timely manner, alerts are filtered and sorted using a strict triaging hierarchy:

* **Rule 1: Operational Filtering:** Target exclusively `New` and unassigned tickets. Analysts must never interfere with baseline tickets already actively marked `In Progress` by other team members.
* **Rule 2: Severity Hierarchy:** Sort incoming payloads from highest to lowest severity values (**Critical ──► High ──► Medium ──► Low**). Critical and High indicators imply direct threat execution sequences.
* **Rule 3: Temporal Sequencing (Time):** If severities match, prioritize the **oldest alert** in the queue. The threat actor in the older instance has had a wider window of time to achieve persistence or initiate data staging protocols.

## 5. Active Lab Walkthrough & Case Analysis

Operating within the TryHackMe SIEM interface, I performed end-to-end triage on three independent telemetry alerts generated across the mock enterprise environment:

### 🛡️ Incident Triage 1: Potential Data Exfiltration (Critical Severity)
* **Telemetry Logs:** Detection rule triggered due to a massive outbound data transmission measuring **5.8 GB**, clearing the standard 5 GB rule limit.
* **Forensic Context:** Parsing fields showed the `Source IP (192.168.45.66)` map to physical host segment `UK04/MEETINGROOM`. The `Destination URL` evaluated to legitimate infrastructure: `*.zoom.us`.
* **Analyst Verdict:** **FALSE POSITIVE**
* **Justification:** Zoom is an authenticated, globally verified corporate tool. High data output originating from an enterprise meeting room indicates standard daily business procedures, specifically an active teleconference or meeting cloud-recording upload. No indicators of compromise detected.
* **Final Status:** `Closed`

### 🛡️ Incident Triage 2: Double-Extension File Creation (High Severity)
* **Telemetry Logs:** A double-extension download vector was flagged executing on host machine `LPT-HR-009` under user account `S.Conway`.
* **Forensic Context:** The download was pulled through `chrome.exe`. The target dropped file was named `cats2025.mp4.exe`. The external host infrastructure resolved to an untrusted domain name: `https://freecatvideoshd.monster/cats2025.mp4.exe`.
* **Analyst Verdict:** **TRUE POSITIVE**
* **Justification:** Valid video data formats deploy within container extensions like `.mp4`. Masquerading an executable utility file (`.exe`) inside a video naming convention is a signature weaponization blueprint used to execute initial access code via social engineering. The `.monster` Top-Level Domain confirms non-corporate infrastructure.
* **Final Status:** `Closed`

### 🛡️ Incident Triage 3: Download from GitHub Repository (Low Severity)
* **Telemetry Logs:** Low severity filter flagged a basic command-line tool pulling down software repository objects directly out of public GitHub zones.
* **Forensic Context:** Connection initiated from endpoint `LPT-IT-063` belonging to user `G.Chandler`. The network source originated within the secure `VPN/DEVELOPERS` environment layer. The remote path targeted: `https://github.com/facebook/react`.
* **Analyst Verdict:** **FALSE POSITIVE**
* **Justification:** Facebook React is a trusted, industry-standard web framework. Because the code pull was executed by a verified system developer inside a dedicated developer workspace, the transaction matches expected job profiles. No threat discovered.
* **Final Status:** `Closed`
